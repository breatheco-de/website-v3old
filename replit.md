# The AI Reskilling Platform

### Overview
The AI Reskilling Platform is a minimalistic Learning Management System (LMS) web application focused on marketing. Its primary purpose is to offer an intuitive interface for career path selection and skill acquisition in AI education. The platform aims to provide accessible, high-quality education globally through a scalable, content-driven architecture, with future integration planned for the 4geeks Breathecode API.

### User Preferences
- Icon library: @tabler/icons-react (NEVER lucide-react)
- Design approach: Marketing-focused landing page
- Card border radius: 0.8rem throughout the platform
- Testing: NEVER use playwright for testing - it takes too much time. User prefers manual verification only.
- Font system: Noto Color Emoji for consistent emoji rendering across all operating systems
- Colors: ONLY semantic tokens - NEVER use hardcoded colors like `bg-blue-500`, `text-red-600`, or arbitrary hex values. Only use semantic classes: `bg-primary`, `text-foreground`, `bg-muted`, etc.
- Video: ALWAYS use the `UniversalVideo` component (`client/src/components/UniversalVideo.tsx`) for ALL video content. NEVER use raw `<video>` tags, `<iframe>` embeds, or other video libraries directly.
- Images: ALWAYS use the `UniversalImage` component (`client/src/components/UniversalImage.tsx`) for ALL image content. Reference images by ID from the centralized registry (`marketing-content/image-registry.json`). NEVER use hardcoded image paths in components. Exception: `HeroSingleColumn` uses `image: { src, alt }` object syntax (not `image_id`) — renders a direct `<img>` tag, with backward compatibility fallback to `UniversalImage` for legacy `image_id` data.
- Image Storage: New images go in `marketing-content/images/` (served at `/marketing-content/images/`). Legacy images remain in `attached_assets/` (served at `/attached_assets/`). The `attached_assets/` folder also contains conversation screenshots which are excluded from the registry scanner and gitignored.
- URL Routing: Use `/en/` prefix for English pages and `/es/` prefix for Spanish pages. NEVER use `/us/` - this is incorrect. Example: `/en/geekforce-career-support` (correct), `/us/geekforce-career-support` (wrong).
- Agent Skills: All project-specific agent skills use the `internal-` prefix (e.g., `internal-image-gallery`). They live in `.agents/skills/` and are automatically loaded when working on related subsystems.

### System Architecture
The platform utilizes React with TypeScript, Vite, Tailwind CSS, shadcn UI, wouter, and TanStack Query for the frontend, and Express for the backend.

**Key Architectural Decisions & Features:**
-   **Design System**: Features a clean, card-based layout with a semantic color system, Lato typography, and `@tabler/icons-react`.
-   **Content Management System (CMS)**: A YAML-based system for marketing teams to manage content, rendered dynamically by a `SectionRenderer` component. Content is indexed by an ORM-like utility (`server/content-index.ts`) on startup. Content types are defined in `marketing-content/content-types.yml` as the single source of truth, accessed via `server/content-types.ts` (backend singleton with `getFolder`, `getType`, `getAllTypes`, `getLabel` helpers) and `/api/content-types` endpoint + `client/src/hooks/useContentTypes.ts` hook (frontend). Singular names (program, page, landing, location) are used throughout application logic; plural forms only for filesystem folders and API URL paths. **URL Pattern Modes**: `url_pattern` supports three formats: (1) **Non-localized** — `{ default: "/landing/:slug" }` for a single URL across all locales with no locale prefix; (2) **Localized shorthand** — a string with `:locale` placeholder (e.g., `/:locale/blog/:category/:slug`) expanded at load time by `normalizeUrlPattern()` into per-locale entries; (3) **Custom per-locale** — a per-locale object (e.g., `{ en: /en/career-programs/:slug, es: /es/programas-de-carrera/:slug }`). The Create Content Type dialog offers all three modes via a segmented control. **Content Type Creation**: `addContentType()` in `server/content-types.ts` writes to `content-types.yml`, creates the directory, and seeds a sample entry subdirectory (`sample-<name>/`) with `_common.yml` + one `<locale>.yml` per configured locale as starter templates.
-   **Site-Wide Settings (`marketing-content/settings.yml`)**: Single source of truth for site-wide configuration. Currently contains `i18n` settings with `default_locale` and `supported_locales` (array of `{code, label}`). Loaded by `server/settings.ts` at startup with caching. Exports: `getSupportedLocales()` (string[]), `getDefaultLocale()`, `getLocaleEntries()` (full objects), `getLocaleLabel(code)`, `resetSettings()`, `updateLocaleSettings()`. Exposed to the client via `GET /api/settings/locales` and `PUT /api/settings/locales`. All server-side locale references use these helpers instead of hardcoded values. Client components (`ContentTypesView`, `VariableDetailModal`) fetch locales from the API with fallback defaults. A dedicated `/private/settings` page (`client/src/pages/SettingsPage.tsx`) provides UI for managing locales (add/remove/set default/save). The Create Content Type dialog links to this page instead of inline locale management.
-   **Template Pages System**: A single generic page template (`client/src/pages/page.tsx`) renders all YAML-based pages including the home page (slug defaults to `"home"` for root routes `/`, `/en/`, `/es/`), supporting `/en/:slug` and `/es/:slug` routes. There is no separate `HomePage.tsx` — the home page is just another template page.
-   **Dynamic Content Type Routing**: `App.tsx` keeps static routes for core types (program, location, page, blog) for instant first-paint, while dynamically generating `<Route>` components for any additional content types (including landing) via `/api/content-types` data. `ContentTypeDetail` (`client/src/pages/ContentTypeDetail.tsx`) accepts any `type: string` — uses legacy API paths for program/location/landing and the generic `/api/content-pages/:contentType/:slug` endpoint for all others. New content types added to `content-types.yml` automatically get frontend routes, API endpoint, sitemap entries, and SSR initial-data injection without code changes. The generic API endpoint resolves dynamic entries, builds `singleEntry`, and returns the same shape as type-specific endpoints. `server/initial-data-middleware.ts` falls back to the generic endpoint for unknown types. `server/sitemap.ts` dynamically includes all non-hardcoded content types.
-   **Internationalization (i18n)**: Supports English and Spanish using `react-i18next`, with browser language detection and a language switcher.
-   **SEO & Performance**: Includes comprehensive meta tags, Open Graph, Twitter Cards, Schema.org JSON-LD, `robots.txt`, dynamic sitemaps, self-hosted WOFF2 fonts, server-side Gzip compression, React component memoization, and native lazy loading. Schema.org JSON-LD is injected server-side. **Hreflang tags**: `<link rel="alternate" hreflang>` tags are injected both server-side (SSR in `ssr-schema.ts` for static pages and all database-backed content types including blog posts/listings, using shared utilities from `server/hreflang.ts`) and client-side (`usePageMeta` hook with `alternates` prop + `useAlternateUrls` hook that fetches from `/api/locale-urls`). Includes `x-default` pointing to the English version.
    -   **Image Performance Optimization**: `SectionPriorityContext` (`client/src/contexts/SectionPriorityContext.tsx`) marks sections within `eager_count` as priority. `UniversalImage` reads this context to auto-apply `fetchpriority="high"`, `loading="eager"`, `decoding="sync"` for above-fold images (non-priority get `fetchpriority="auto"`, `loading="lazy"`, `decoding="async"`). SSR preload hints: `resolvePreloadHints()` in `initial-data-middleware.ts` extracts image IDs and direct `src` URLs from priority sections, resolves them via image registry, and `buildPreloadTags()`/`injectPreloadTags()` inject `<link rel="preload" as="image">` into `<head>` in both dev and prod SSR paths. Image registry schema (`shared/schema.ts`) supports `width`, `height`, `preset[]`, `widths_generated[]`, `format`, `srcset[]`, `sizes`, and `parentId` (for derived/cropped variants) for responsive images.
    -   **UniversalImage URL Fallback**: `UniversalImage` builds a `useMemo` reverse index (`src → id`) from the registry. When an image is referenced by its raw URL/path instead of its registry ID, the component resolves it via this index and logs a dev-mode warning with the suggested ID.
    -   **Crop-Resize Derived Image Tracking**: The `POST /api/image-registry/crop-resize` route now stores `parentId` on newly-created variants, and immediately triggers background srcset optimization for the derived image via `processImageFromSrc`. MediaGallery hides derived images by default; a "Derived" badge toggle shows them. The `ImagePickerDialog` crop panel shows a "Size Preset" dropdown with suggestions from the rendering preset's widths/aspect-ratio and the image's actual DOM dimensions, auto-filling the width/height fields.
    -   **Hero Image Fixer + Auto-Optimization**: The `hero-image-tags` fixer (`scripts/validation/fixers/hero-image-tags.ts`) now triggers background `processImageFromSrc` optimization for every image it newly tags with "hero", using the same pattern as the `image-optimization` fixer.
    -   **Image Optimization Pipeline**: `server/image-optimizer.ts` is the shared core module for image processing (Sharp resize, WebP/AVIF conversion, GCS upload of variants). Exports `processImageBuffer()` (from in-memory buffer) and `processImageFromSrc()` (downloads then processes). Used by: (1) `scripts/backfill-images.ts` for batch processing existing images, and (2) `uploadAndRegister()` in `media-gallery.ts` which fires `optimizeInBackground()` after each new image upload — the upload returns immediately while variants are generated asynchronously, updating the registry entry with `width`, `height`, `preset`, `widths_generated`, `format`, and `srcset` upon completion. Only raster images (.png, .jpg, .jpeg, .webp, .avif) are optimized; SVGs and GIFs are skipped.
    -   **Two-Tier Section Loading**: `SectionRenderer` uses a two-tier strategy: root-level `settings.loading.eager_count` (default 3) + per-section `load: eager|lazy` override. Eager sections render immediately; lazy sections are deferred via `IntersectionObserver` (`DeferredSection` with 200px rootMargin). All section components are eagerly imported (no code-splitting); "lazy" means deferred rendering only. Resolution: explicit YAML `load` > component `section_defaults` from schema.yml > position-based (`index < eager_count`) > lazy. Components can declare default section properties via `section_defaults` in their `schema.yml` (e.g., `section_defaults: { load: eager }` for `sticky_cta`); `getComponentSectionDefaults()` in `server/component-registry.ts` builds a cached lookup, and `applyComponentSectionDefaults()` injects any missing defaults server-side before sending sections to the frontend. The `section_defaults` mechanism is generic — any section-level property can have a default, not just `load`. The `components` validator checks that `section_defaults.load` values are valid.
    -   **Initial Data Inlining**: `server/initial-data-middleware.ts` intercepts HTML responses, resolves the URL via `contentIndex.resolveUrl`, loads page data server-side, and embeds it as `<script id="__INITIAL_DATA__">`. On the client, `client/src/lib/initialData.ts` hydrates this into React Query cache before render, eliminating the initial API fetch.
-   **URL Redirects System**: Handles 301 redirects defined in YAML meta properties and `custom-redirects.yml`, with validation and support for regex patterns.
-   **Content Page Types**: `CareerProgram`, `LandingPage`, `LocationPage`, and `TemplatePage` are defined as plain TypeScript interfaces in `shared/schema.ts` (not Zod schemas). Content is loaded without schema validation — `loadContent()` returns raw merged YAML data cast to the interface type. Meta fields (`page_title`, `description`) are optional; `applyMetaFallback()` in `server/routes.ts` fills missing `page_title` from `title` and sets `description` to empty string at render time. The diagnostics endpoint surfaces missing meta as `MISSING_META` warnings (non-blocking) instead of hard failures. The `content-editor.ts` no longer performs post-edit schema validation for these types.
-   **Versioned Component Registry**: A filesystem-based registry at `marketing-content/component-registry/` stores versioned component schemas and examples, with Zod schemas defined within each component's registry folder.
-   **SSR (Server-Side Rendering)**: Production SSR uses a Vite-built SSR bundle at `dist/server/entry-server.js`. Browser-only components (`ChatWidget`, `DebugBubble`, `VariableModalHost`) are wrapped in `<ClientOnly>` in `App.tsx`. Third-party libraries with CJS/ESM interop issues (e.g., `react-fast-marquee`) must be imported through wrapper modules in `client/src/lib/` that handle double-wrapped default exports. An SSR smoke test script is available at `scripts/ssr-check.sh`.
-   **Session Management System**: A hybrid client-side system providing IP-based geolocation, nearest campus calculation, UTM tracking, and language detection.
-   **A/B Testing Experiment System**: A cookie-based system for content variants, supporting various targeting variables and a comprehensive `Experiment Editor`.
-   **UniversalVideo Component**: Mandatory component (`client/src/components/UniversalVideo.tsx`) for all video content.
-   **UniversalImage Component**: Mandatory component (`client/src/components/UniversalImage.tsx`) for all image content, referencing `image-registry.json`.
-   **ContentIndex Singleton**: `server/content-index.ts` is the single source of truth for all YAML content read/write operations. It provides `safeYamlLoad` (template-variable-safe YAML parsing), `loadLocaleData` (locale file only, no `_common.yml` merge — use for writes; returns `isSharedTemplate: true` when fallback to shared template is used), `loadMergedContent` (merged with `_common.yml` — use for validation/display; also returns `isSharedTemplate`), `loadCommonData`, `loadContent` (merged loading with optional schema validation — schema param is optional; when omitted, returns raw merged data cast to the generic type), `getContentFilePath`, `getContentFolderPath`, `getCommonFilePath`, `isDatabaseBacked` (checks if content type has `database.slug` configured), `listContentSlugs`, `getAvailableLocalesOrVariants`, `resolveUrl` (resolves a URL path to `{contentType, slug, entry, fromDatabase?, params?}` using URL patterns from `content-types.yml`; returns `fromDatabase: true` when a database-backed type has no disk entry), `isKnownUrl` (delegates to `resolveUrl`), `getCommonFields(contentType)` (returns `CommonFieldInfo` with `common` and `partial` dot-path properties across all entries — lazily computed and cached), and `invalidateCommonFields(contentType)` (clears the cache for a type so it's recomputed on next access; called automatically when field mappings are saved via PUT config). Supports multi-parameter URL patterns (e.g., `/en/blog/:category/:slug`) extracting all named params. Auto-creates `_common.single.yml` and `single.{locale}.yml` templates on startup for database-backed content types only. **Shared Template Fallback**: For database-backed content types, `getContentFilePath` falls back to `single.{locale}.yml` when no per-slug YAML file exists; `getContentFolderPath` falls back to the type root directory; `loadCommonData` falls back to the type root's `_common.single.yml`. This enables section editing on database-backed entries (e.g., blog posts) — edits modify the shared template, affecting all entries of that type. The `content-editor.ts` skips strict schema validation for database-backed types since their templates use Jinja-style variables. All content file read/write operations in `content-editor.ts`, `bindings.ts`, `sitemap.ts`, and `routes.ts` use these methods to avoid duplicating `_common.yml` merge logic, ensure safe template variable handling, and prevent accidental content duplication in locale files. NEVER use raw `yaml.load()` for content files — always use `contentIndex.safeYamlLoad()` or the higher-level methods.
-   **Inline Editing System**: A capability-based system for human editors and AI agents, allowing direct content modification on the site via `EditModeContext` and server-side APIs. Two separate endpoints: `/api/content/edit-sections` (writes to locale files for section operations) and `/api/content/edit-common` (writes to `_common.yml` for meta, slug, schema, locations). This ensures locale files only contain locale-specific overrides. **Variable Type Chooser**: On shared template pages (database-backed content types), the "Convert to variable" flow first opens a `VariableTypeChooserModal` asking whether to create a "single" (entry field) or "global" variable. Single variables use `{{ single.fieldname | fallback }}` syntax and are picked from the content type's `field_mapping` via `SingleVariablePickerModal`. Global variables use `{{ global.varname | fallback }}` syntax and go through the existing `VariableDetailModal`. On non-shared-template pages, the global modal opens directly. New global variables always enforce the `global.` prefix. **Variable Manager**: `client/src/lib/variable-manager.ts` (renamed from `variable-resolver.ts`) is the unified client-side variable resolution system. It handles both `global.*` and `single.*` variables in a single `resolveDeep` pass. Single vars are resolved using `singleEntry` data passed from the server (database record fields via `TemplatePage.singleEntry`), while global vars use `definitions` from `/api/variables`. Server-side `single-resolver.ts` is no longer called during page load — all resolution is client-side. The `resolveDeep` options accept `{ preserveTemplate?: boolean; singleEntry?: Record<string, unknown> }`. This ensures both variable types persist as template syntax in YAML and are only resolved for preview display.
-   **Theme Configuration System**: A centralized `marketing-content/theme.json` defines allowed colors for backgrounds, accents, and text.
-   **Validation System**: A modular framework (`scripts/validation/`) for both preventive (API calls) and reactive (CLI batch scanning) validation across redirects, meta, schema, sitemap, components, theme compliance, database-single integrity (conflict detection, template validation), and slug/URL conflict detection across all non-DB content types. **Slug Conflict Validation**: Creation-time 409 errors prevent duplicate slugs within the same content type. The `slug-conflicts` diagnostics validator (`scripts/validation/validators/slug-conflicts.ts`) scans all non-DB content types for URL collisions — two entries producing the same resolved URL — and reports them as errors.
-   **GitHub Content Sync**: Allows content edits made via the inline editor to be committed back to a GitHub repository. Includes an **Auto-Commit System** (`server/auto-commit.ts`) that automatically commits marketing-content file changes to GitHub on a configurable throttled schedule (default 5s). Uses Tree API for batch commits with Contents API fallback for conflict resolution. Features exponential backoff retry (5s→10s→30s→60s cap) on failures with automatic reset on success. Sync state is persisted to GCS bucket (`sync/sync-state.json`) to survive deployments. API endpoints at `/api/github/auto-commit/*` for status, flush, config, and conflict management. Status endpoint exposes `nextSyncAt`, `isCommitting`, `githubConfigured`, and per-file `pendingFilesDetails` (author, timestamp). The DebugBubble sync modal shows live countdown timer, auto-commit queue with per-file author/timestamp, conflict resolution UI, and a collapsible "Manual Actions" section for legacy sync operations. Includes a **Sync Log** (`server/sync-log.ts`) that records all GitHub sync activity (RESTART, RECONCILE, WEBHOOK, AUTO-PULL, COMMIT, CONFLICT, ERROR) with timestamps, commit SHAs, and instance IDs to `.sync-log-state.txt`, persisted to GCS bucket. Each server restart logs instance ID, env, and PID to detect Replit redeployments. API endpoints at `/api/github/sync-log` (full text log) and `/api/github/sync-info` (structured status with webhook health, instance info, recent entries). A **SyncStatusPopover** component in the DebugBubble shows webhook status, instance ID, recent log entries, and an "Open Full Log" button when clicking the sync status indicator. **Webhook System** (`server/github.ts` `ensureWebhook`): auto-creates GitHub push webhooks on startup with HMAC-SHA256 signature verification, stores webhook config in sync state, and auto-pulls non-conflicting incoming files via `/api/github/webhook` endpoint.
-   **AI Agent Per-Mode Model Selection**: `llm.yml` uses a nested `model: { default, chat }` structure. `resolveModel(config, mode)` in `AgentService.ts` resolves models per mode: chat conversations use `LLM_CHAT_MODEL` env → `model.chat` → `model.default`, background tasks (tagging, clustering) use `LLM_MODEL` env → `model.default`. Backward-compatible with flat `model: string` format. Admin UI at `/private/ai-knowledge` includes a "Models" dialog for configuring both values. PATCH `/api/admin/ai/knowledge` accepts `model_default`/`model_chat` fields, writes nested object to `llm.yml`, and hot-reloads the agent.
-   **Conversion Tracking System**: A centralized module (`client/src/lib/tracking.ts`) for type-safe analytics and conversion tracking via Google Tag Manager (GTM).
-   **Content Diagnostics System**: A self-service diagnostics page at `/private/diagnostics` providing global health and per-page analysis.
-   **Media Module**: A pluggable storage provider system (`server/media/`) supporting local filesystem and cloud storage (Google Cloud Storage) using a strategy pattern.
-   **MediaGallery Singleton**: Consolidates all image registry operations and API endpoints (`server/media-gallery.ts`), including upload and migration functionalities.
-   **FAQ Per-Item Location Visibility**: FAQ sections support `item_overrides` for hiding specific FAQ items based on the visitor's location.
-   **Section Bindings System**: `BindingManager` singleton (`server/bindings.ts`) synchronizes content between sections of the same component type and locale across different pages, with CRUD operations, immediate content propagation on bind, optional group naming, and automatic propagation on edit. Uses stable `sectionId` strings (stored in YAML as `section_id`) instead of fragile numeric indices — `ensureSectionId()` auto-generates IDs when binding, `resolveSectionIndex()` resolves position at runtime. API routes accept `sectionIndex` from the frontend and resolve to `sectionId` internally; responses include both. Legacy `sectionIndex`-based bindings are auto-migrated on server load.
-   **Unified Field Mapping & Indexes**: `field_mapping` and `indexes` are **top-level** properties on every content type in `content-types.yml` — available to ALL types (DB and non-DB). The `database` block retains only `slug`. For DB types, `field_mapping` maps DB columns → content fields (with `_slug`/`_locale` special keys). For non-DB types, it exposes YAML keys from the merged content chain (`_common.single.yml` → `_common.yml` → `[locale].yml`) as `{{ single.* }}` template variables. The server builds a `singleEntry` context via `buildSingleEntryFromContent()` in `routes.ts`. `indexes` enables filtering when listing entries. Helper functions all read from `entry.field_mapping`/`entry.indexes` (not `database.*`). Schema:
    ```yaml
    blog:
      field_mapping:               # top-level, not inside database
        _slug: slug                # special: DB identifier field
        _locale: lang              # special: DB locale field
        title: title
        content: content
      indexes: [status, category]  # top-level
      database:
        slug: blog_posts           # only DB source name

    program:                       # non-DB type
      field_mapping:
        slug: slug
        title: title
    ```
    Underscore-prefixed keys (`_slug`, `_locale`) in `field_mapping` are mandatory special fields for DB-backed types only. `_locale` identifies the locale field (string or object `{ source, default }`). `_slug` identifies the unique identifier field. Both are excluded from `getFieldMapping()` but included in `getFullFieldMapping()`. Helper functions all read from `entry.field_mapping` and `entry.indexes` (not `database.*`): `getDatabaseName(type)` reads `database.slug`, `getFieldMapping(type)` reads `entry.field_mapping` excluding `_`-prefixed keys, `getFullFieldMapping(type)` includes all, `getLocaleKey(type)` reads `field_mapping._locale`, `getSlugField(type)` reads `field_mapping._slug`, `getIndexes(type)` returns `entry.indexes || []`, `getDatabaseConfig(type)` returns the `database` object (only `slug`). `updateContentTypeConfig` saves with header comments preserved via `writeConfigWithHeader()`. When configured, the content type uses shared `single.{locale}.yml` templates (auto-created on startup for DB types) with `{{ single.fieldname }}` template variables resolved client-side via `variable-manager.ts`. API routes: `GET /api/database-single/:contentType/:slug?locale=en` (SSR), plus generic CRUD at `/api/content-types/:type/config`, `/api/content-types/:type/items`. Frontend: `DatabaseSinglePage` component renders DB-backed items; all page components pass `singleEntry` to `SectionRenderer`. Management UI at `/private/type/:contentType` (`ContentTypeManagePage.tsx`).
    Every content type has two potential sources displayed in the UI: **Static Entries** (subfolders in `marketing-content/{folder}/` with locale YMLs — always present) and **DB Entries** (dynamic items from a connected database — optional). The `ContentTypeManagePage` header shows both with counts. The `ContentTypesView` in the DebugPanel also shows both sources per type. The `/api/content-types` and `/api/content-types/:type/config` endpoints include `static_entry_count`.
    **Field Mapping Transformers (`function:` prefix)**: Field mapping values can use a `function:` prefix followed by a base64-encoded JS function body for computed/transformed values. Format: `fieldName: "function:<base64>"`. The function signature is `(value, item) => result` where `value` is `item[targetKey]` and `item` is the full raw record. Functions run in a sandboxed `vm.runInNewContext()` with 50ms timeout and no access to Node globals/modules. Utility module: `server/transform.ts` exports `FUNCTION_PREFIX`, `isTransformer()`, `extractFunctionBody()`, `compileTransformer()` (with caching), `runTransformer()`, `resolveFieldValue()` (unified resolver for both dot-paths and transformers), `applyTransformIfNeeded()` (convenience for locale normalization points), and `getValueByPath()`. Used in `applyContentTypeMapping` (database.ts), inline mapping in routes.ts, and locale normalization across routes.ts/dynamic-entries.ts/sitemap.ts. The blog content type uses a transformer on the `lang` field to normalize `us→en` locale values. `getLocaleSource(type)` in `content-types.ts` returns the raw `_locale` mapping value (including `function:` prefix if transformer). Both UIs (ContentTypeManagePage DataSourceDialog and PrivateDatabases FieldMappingEditor) support toggling fields to transformer mode via an `IconCode` button, showing a monospace textarea for JS function editing. Values are decoded from base64 for display and re-encoded on save.
-   **Blog System**: The blog is a fully generic database-backed content type — there is no blog-specific server code. Configuration is centralized in `content-types.yml` under the `blog` entry with `database.slug` and top-level `field_mapping` (including `_locale: lang` for locale-aware lookups). Blog posts are fetched via `databaseManager.fetchMappedItems("blog")`. Blog single posts are rendered via the Database-Backed Content Type system using `marketing-content/blog/single.{locale}.yml` templates. SSR (meta tags, schema.org, hreflang) uses the generic `generateDatabaseSsrHtml`/`generateListingSsrHtml` from `server/ssr-schema.ts`. Markdown content is fetched via utilities in `server/markdown.ts`. Management UI at `/private/type/blog` (generic `ContentTypeManagePage`). Legacy `/private/blog` route still renders the old `BlogManagePage` for backward compatibility. Frontend routes: `/en/blog` and `/es/blog` now use `TemplatePage` (YAML-driven via `marketing-content/pages/blog/{en,es}.yml` with `listing_cards` component and `dynamic_entries`); `/en/blog/*/*` and `/es/blog/*/*` use `DatabaseSinglePage`.
-   **Dynamic Entries System**: A server-side resolution system (`server/dynamic-entries.ts`) that allows any list-based YAML component to pull items from a content type or database. Sections declare `dynamic_entries: { content_type, database, filter, limit, sort, item_template, hardcoded_entries }`. `item_template` maps DB fields to component item fields using `{{ single.* }}` syntax; `hardcoded_entries` adds static items. Both are nested under `dynamic_entries` since they only exist in that context. The resolver (`resolveDynamicEntries(sections, locale)`) runs server-side in the page data loading pipeline (both `/api/pages/:slug` and `initial-data-middleware.ts` for SSR). It fetches items via `databaseManager.fetchMappedItems()`, filters by locale, applies sort/limit, resolves template variables, and merges with `hardcoded_entries`. **Computed fields**: For content types with a `url_pattern`, the resolver injects `_resolved_url` on each item (use as `{{ single._resolved_url }}`). The underscore prefix signals a computed field, not a raw DB column. Output preserves `dynamic_entries` in the API response alongside resolved `items` + `_dynamic_meta` — the editor sees both authored YAML and resolved data. The content editor (`content-editor.ts`) strips transient `items` and `_dynamic_meta` before saving sections with `dynamic_entries`. Components stay "dumb" and just render `items`. Components that support dynamic entries declare `dynamic_item_fields` in their `schema.yml` (e.g., `listing_cards`). Zod schema for `listing_cards` is in `shared/schema.ts` (`listingCardsSectionSchema`). Backward compat: top-level `item_template`/`hardcoded_entries` are still accepted by both the resolver and Zod schema.
-   **Generic URL Pattern Resolver**: `resolveUrlPatternWithMapping(pattern, record, locale, fieldMapping?)` in `server/content-types.ts` dynamically resolves URL patterns (e.g., `/en/blog/:category/:slug`) using `field_mapping` from `content-types.yml`. For each `:param` token, it checks if `fieldMapping[param]` exists and reads the mapped source field from the raw record. Falls back to `record[param]` if no mapping. Handles objects with `.slug` property, arrays (joined with commas), and collapses double slashes. Convenience function `resolveContentTypeUrl(type, record, locale)` auto-loads the url_pattern and field_mapping. **Important**: Raw DB records (from `.cache/db-*.json`) need the `fieldMapping` to resolve params correctly (e.g., `:category` → `cluster_slug`). Already-mapped records (after `applyContentTypeMapping` in `database.ts`) have fields like `category: { slug: "..." }` and should pass `null` for fieldMapping. The sitemap uses raw records with mapping; SSR uses mapped records without mapping.
-   **Generic Database System**: A configurable data collection engine (`server/database.ts`) that serves normalized items from external sources (API, CSV, YAML). Databases are defined at `marketing-content/db/<name>/config.yml` with `source` (type + connection details), `cache` (TTL), and `field_mapping` (dot-notation path mapping). The engine provides `listDatabases()`, `loadConfig()`, `saveConfig()`, `fetchDatabaseItems()` (with memory + file cache), and `testDatabaseSource()`. API routes at `/api/databases[/:name][/items|/refresh|/test|/raw-fields|/analyze-fields]`. Management UI at `/private/databases` with list and detail views (data table with sort/search, AI-powered field mapping editor). Database names are validated against `[a-z0-9_-]+` to prevent path traversal. Uses a `DatabaseManager` singleton class (`databaseManager` export) — configs are loaded into memory on startup, with `list()`, `get()`, `create()`, `update()`, `delete()`, `fetchItems()`, `test()`, `warmup()`, `reload()`, `getCacheInfo()`, `getRawItems()`, `getRawFields()`, and `getFieldCount()` methods. A "Create Database" dialog in the UI supports multi-step creation: basics, source config, test connection with auto-generated field mapping from sample response.
    **Dual Cache System**: `fetchItems()` saves two cache files: `.cache/db-{name}.json` (mapped data) and `.cache/db-{name}-raw.json` (original raw API data). The detail endpoint (`GET /api/databases/:name`) includes `cache_status: { fetched_at, item_count }` for showing KPI data without manual fetch.
    **Two-Layer Field Mapping**: There are two distinct mapping layers: (1) **Raw API → Database** mapping defined in `marketing-content/db/<name>/config.yml` transforms raw API fields into normalized database fields before caching (uses AI auto-detect via `POST /api/databases/:name/analyze-fields`); (2) **Database → Content Type** mapping defined in `content-types.yml` under top-level `field_mapping` maps normalized database fields to content type properties. Content-type field mappings support `raw.fieldName` prefix to reference original API data and `db.fieldName` (or no prefix, default) for mapped database fields.
    **Cache Warmup**: On server startup (after `listen`), `databaseManager.warmup()` is called fire-and-forget. It checks each database's file cache — if missing or TTL-expired, it pre-fetches from the external source so no visitor hits a cold cache.
-   **DebugBubble Modular Architecture**: The `client/src/components/DebugBubble/` directory is a modular component system. `index.tsx` is the orchestrator (state management + composition only, ~1,900 lines). All rendering logic for views and modals lives in separate files under `components/`. NEVER add new inline view/modal JSX to `index.tsx` - always create a new file in `components/` and import it. Shared types live in `types.ts`, shared helpers in `utils/`. The main panel content (auth views, menu views, session controls) is in `DebugPanelContent.tsx`. On desktop, the panel renders inside a Popover; on mobile (`useIsMobile` at 768px), it renders inside a full-screen Sheet (`side="bottom" h-[100dvh]`).
-   **Content-Type-Wide Defaults (`_common.single.yml`)**: Each content type directory can contain a `_common.single.yml` file that defines content-type-wide defaults (meta, schema, section_defaults). Merge priority: `_common.single.yml` (content-type-wide) → `_common.yml` (per-slug) → `[locale].yml` (locale-specific). Both `loadDatabaseSinglePage` and the regular content loading paths merge this as the base layer. API endpoints: `GET /api/content-type/:name/single-defaults` (read) and `PUT /api/content-type/:name/single-defaults` (write/merge).
-   **Section X Spacing & Max Width (`paddingX`/`marginX`/`maxWidth`)**: Sections support horizontal padding, margin, and max-width constraint via `paddingX`, `marginX`, and `maxWidth` in `sectionLayoutSchema`, using the same `ResponsiveSpacing` schema as vertical spacing (`{ mobile?, desktop? }`). Padding/margin values use composite shorthand (`"left right"`) or single value. Max-width uses preset keys: `none`, `sm` (672px), `md` (768px), `lg` (896px), `xl` (1152px), `2xl` (1280px), `full` (100%). CSS variables: `--section-pl/pr/ml/mr/mw-mobile/desktop`. When maxWidth is active, the section is auto-centered with `margin-inline: auto` (unless explicit marginX is set). An `IconSpacingHorizontal` button in the `EditableSection` toolbar opens a popover with desktop/mobile breakpoint toggle and preset buttons for max width, padding left/right, and margin left/right. After saving, if no default exists in `_common.single.yml`, prompts to save as content-type-wide default.
-   **Content Creation Locale Exclusion**: The `CreateContentModal` supports excluding locales from content creation. Trashcan icons next to each locale row toggle exclusion; at least one locale must remain active. Excluded locales are sent as `skipLocales` in the POST body to `/api/content/create`, which skips writing those locale files. The file preview updates dynamically to reflect which files will be created.
-   **Duplicate with Content Type Change**: When duplicating a content entry via `CreateContentModal`, the content type selector is always visible (pre-selected to the source entry's type). Users can choose a different non-database-backed target type. All non-DB content types (including landing) support cross-type duplication since they share the same `_common.yml` + `{locale}.yml` file structure. When a different type is selected, a warning banner appears with a "Read more" link to a detail modal explaining the conversion. The backend `ContentIndex.duplicateWithTypeChange()` method handles conversion: replaces `{{ single.* }}` template variables with resolved fallback values (preserving those inside `dynamic_entries.item_template`), removes source-type-specific `field_mapping` properties, strips `section_id` from all sections, and skips `_common.single.yml`/`single.*.yml` files. The route handler at `/api/content/create` dispatches to this method when `changeContentType: true` is in the request body.
-   **Layout Menu System (`layout.menu`)**: Controls which navbar (top) and footer (bottom) menus render per content type and per page. Configured in `content-types.yml` via `layout.menu.top` and `layout.menu.bottom` (referencing menu IDs from `marketing-content/menus/`). System default when `layout` is absent: `{ menu: { top: null, bottom: null } }` — new content types have no menus. Existing core types (`program`, `location`, `page`, `blog`, `downloadable`) declare `top: main-navbar`, `bottom: main-footer`; `landing` declares `null`/`null`. Per-entry overrides in `_common.yml` or locale files take precedence. Resolution order: locale file → `_common.yml` → `_common.single.yml` → `content-types.yml` → system default. Backend: `getLayout(type)` and `resolveLayout(contentType, mergedData)` in `server/content-types.ts`; `listAvailableMenus()` scans menu files. All page API endpoints inject resolved `layout` into responses. Frontend: `ContentTypeDetail`, `page.tsx`, `DatabaseSinglePage` read `data.layout.menu` and conditionally render `<Header menuId={top} />` (eager) and `<Footer menuId={bottom} />` (lazy via `LazyRender` with `IntersectionObserver`). `Header` and `Footer` accept optional `menuId` prop (defaults to `main-navbar`/`main-footer`). Edit mode: `MenuSlotPlaceholder` component shows dashed placeholder for empty slots. For assigned menus, it renders a floating toolbar (absolutely positioned, hidden until hover via `opacity-0 group-hover:opacity-100`) with three buttons: pencil+name (navigates to `/private/menu-editor/<menuId>?locale=<locale>`), swap (popover to change menu), and trash (clear menu). Page components wrap Header+placeholder and Footer+placeholder in `group relative` containers for hover detection. **Scope confirmation**: When adding/removing/swapping a menu via the floating toolbar, a Dialog asks whether to apply the change to all entries of that content type (`PUT /api/content-types/:type/layout` — updates `content-types.yml` and cleans up per-entry overrides) or this entry only (`editCommonContent` — writes to `_common.yml`). **Menu usage tracking**: The `contentIndex` scan builds a `menuUsage` map (alongside `imageUsage` and `variableUsage`) tracking which YAML files reference each menu ID. Exposed via `getMenuUsageByMenuId(menuId)`. API: `GET /api/menus/:name/usage` returns `{ defaultContentTypes, overrides }` — defaults from `content-types.yml`, overrides from per-entry files. **MenuEditor info box**: `MenuUsageInfo` component in `MenuEditor.tsx` shows which content types use the menu by default, with a clickable "N additional URLs" popover listing per-entry overrides.
-   **Per-Locale Content Deletion**: The `DeletePageModal` supports per-locale deletion. When `availableLocales` is provided, one checkbox per locale is shown. The current locale is pre-checked; others are unchecked by default. Users check/uncheck individual locales to control exactly which `.yml` files get deleted. If all locale files are removed, the entire folder (including `_common.yml`) is cleaned up. The modal is reused in both the DebugBubble and `ContentTypeManagePage` (static entries dropdown).

-   **PrivatePreview dynamic content type support**: `PrivatePreview.tsx` now dynamically supports all content types (including database-backed ones like `blog`) instead of hardcoding only 4 types. It fetches content types from `/api/content-types` and routes DB-backed types through `/api/content-pages/:type/:slug` (which handles both static and DB types with full dynamic entry resolution). The hardcoded `STATIC_API_PATHS` map is kept as a fallback for the original 4 types (`program`, `landing`, `location`, `page`). Blog entries are now accessible via `/private/preview/blog/<slug>?locale=en`.
-   **DatabaseSinglePage menu editing**: `DatabaseSinglePage.tsx` now includes `MenuSlotPlaceholder` wrappers (same pattern as `page.tsx` and `ContentTypeDetail.tsx`) with `isSharedTemplate` flag. When `isSharedTemplate` is true, `MenuSlotPlaceholder` skips the scope dialog and directly applies changes to the content type default (since DB-backed types don't have per-entry `_common.yml` files).
-   **Visual Theme Editor (`/private/theme-editor`)**: A split-pane UI for live-editing the site's design tokens. Left panel: Light/Dark mode toggle, per-token HSL sliders (H/S/L range inputs with gradient tracks + hex input) for all semantic CSS variables (`--primary`, `--secondary`, `--accent`, `--background`, `--muted`, `--card`, `--border`, `--radius`, etc.), radius slider with preset buttons, Save/Reset. Right panel: "Atoms" tab (filtered `molecules.json` entries tagged `theme-preview`, grouped by button/badge/form/card/stats/banner, rendered via `MoleculeRenderer.tsx`) + "Combinations" tab (iframes from `/api/component-registry`). Backend: `GET /api/theme` returns `marketing-content/theme.json` (includes `colors.light`, `colors.dark` HSL maps), `PUT /api/theme/colors` persists color overrides. `server/initial-data-middleware.ts` → `buildThemeCssOverrides()` reads `theme.json` and injects `<style id="__theme_overrides__">` with `:root { }` + `.dark { }` CSS variable blocks before `</head>` on every HTML response. Accessible via the DebugBubble "Theme Editor" menu item.
